# d34 — AI-ассистент, который работает с файлами

Ассистент получает **цель на естественном языке**, самостоятельно исследует
структуру проекта и через локальный MCP-сервер читает, ищет, проверяет, создаёт и
изменяет файлы. Основа решения — tool-use цикл из `d31`, расширенный с пассивного
RAG-поиска до реальных файловых операций.

## Что умеет

- `list_files` — изучить структуру проекта;
- `search_text` — найти все употребления компонента/API в нескольких файлах;
- `read_files` — прочитать до 10 связанных файлов с номерами строк;
- `write_file` — атомарно создать или заменить UTF-8 файл;
- `check_invariants` — проверить пробелы, табы, финальный перевод строки и
  повторяющиеся Markdown-заголовки;
- `diff_changes` — показать единый diff всех изменений текущего запуска.

Ассистенту явно предписано начинать с обзора, исследовать минимум 2–3 файла,
проверять изменения и завершать работу diff. Все пути изолированы внутри
переданного корня проекта; `.git`, `.venv`, `node_modules`, сборочные каталоги и
файлы больше 1 МБ недоступны.

## Установка

```bash
cd /Users/viktor/ai_ac/d34
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
```

Добавьте `ANTHROPIC_API_KEY` в `.env`.

## Сценарий 1: найти все использования API

Задача сформулирована на уровне цели — имена файлов ассистент находит сам:

```bash
python assistant.py example_project \
  "Найди все места, где используется PaymentAPI, и объясни роль каждого использования"
```

### Интерактивный режим

Чтобы выбрать проект один раз, запустите ассистента без цели:

```bash
python assistant.py example_project
```

После этого вводите разные задачи в одной консольной сессии:

```text
d34> Найди все использования PaymentAPI
d34> Проверь, соответствует ли документация реализации
d34> Создай ADR о текущем устройстве платежей и покажи diff
d34> /exit
```

Каждая цель выполняется как отдельная файловая операция: ассистент заново проверяет
актуальное состояние проекта, поэтому следующая команда видит изменения предыдущей.

Ассистент просматривает структуру, ищет `PaymentAPI`, читает реализацию,
потребителей и документацию. Файлы не меняются; в результате выдаётся список с
путями и строками.

## Сценарий 2: обновить документацию по коду

```bash
python assistant.py example_project \
  "Проверь реализацию и все использования PaymentAPI, затем приведи документацию API в соответствие с кодом"
```

Ассистент анализирует минимум `src/payments.py`, `src/orders.py`,
`src/refunds.py` и `docs/api.md`, при необходимости сохраняет документацию и
показывает unified diff. Повторный запуск не дублирует текст и сообщает, что файл
уже актуален.

## Сценарий 3: проверить инварианты и исправить нарушения

```bash
python assistant.py example_project \
  "Проверь текстовые файлы проекта на нарушения инвариантов, исправь безопасные нарушения и покажи diff"
```

## Воспроизводимая проверка

Тесты не используют Claude API. Они запускают настоящий MCP stdio round-trip и
проверяют работу с тремя файлами, поиск, чтение, защиту путей, инварианты,
идемпотентную запись и diff:

```bash
pytest -q
```

Пример `example_project` намеренно хранится в готовом согласованном состоянии.
Чтобы гарантированно увидеть diff, измените цель так, чтобы она попросила создать
новый ADR или changelog; например:

```bash
python assistant.py example_project \
  "Проанализируй устройство платежей и создай ADR с описанием текущего решения"
```

## Состав решения

- `assistant.py` — CLI;
- `agent.py` — цикл Claude с вызовами инструментов;
- `mcp_client.py` — MCP-клиент;
- `mcp_server.py` — безопасные реальные операции с файлами;
- `example_project/` — проект из пяти файлов для демонстрации;
- `tests/` — воспроизводимые offline-тесты.
