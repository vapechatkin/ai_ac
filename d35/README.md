# d35 — SnackPause

Мобильный AI-ассистент осознанных перекусов для Android и iOS. Пользователь
описывает, что хочет съесть или выпить, а ассистент коротко уточняет контекст и
помогает отличить физический голод от импульса, привычки, жажды или эмоционального
триггера.

Цель приложения — уменьшать переедание и количество неосознанных перекусов, не
запрещая еду и не создавая чувство вины. Если пользователь действительно голоден,
ассистент прямо рекомендует поесть.

## Как AI участвует в процессе

Приложение использует облачную модель **Alice AI LLM Flash** через
OpenAI-совместимый API Yandex AI Studio. Один большой диалог разбит на компактный
пайплайн:

```text
Сообщение пользователя
        ↓
Intake Agent → CasePlan + первый уточняющий вопрос
        ↓
Interview Agent ↔ пользователь (по одному полезному вопросу)
        ↓
Advice Agent → кандидат рекомендации
        ↓
Review Agent → проверенная финальная рекомендация
        ↓
Memory Agent → короткая запись эпизода в локальную память
```

Между AI-вызовами работают детерминированные валидаторы. Они проверяют структуру,
не дают повторять вопросы, отбрасывают пустые ответы и не позволяют сохранять
слабую гипотезу как подтверждённый факт. Оркестратор реализован обычным кодом:
отдельная модель для управления пайплайном не расходует токены.

Advice Agent и Review Agent образуют цикл редактирования: если автоматическая
проверка отклоняет черновик, Review Agent получает конкретную причину и
переписывает рекомендацию. Для нестабильного JSON и временных сетевых ошибок есть
ограниченные повторы, а пользователь получает понятное сообщение вместо
технического исключения.

## Память и RAG

История диалогов, эпизоды и факты хранятся только на устройстве в SQLite. После
завершения сессии Memory Agent формирует компактную запись, а Yandex Text
Embeddings v2 создаёт для неё вектор. Перед новым диалогом приложение выбирает не
всю историю, а максимум три похожих эпизода с cosine similarity не ниже `0.55`.
Если embedding получить не удалось, RAG возвращает пустую историю, а не последний
эпизод.

RAG никогда не передаётся Intake Agent, Interview Agent или Memory Agent и не
может заполнять факты текущей сессии. Прошлые эпизоды доступны только Advice Agent
и Review Agent как слабый контекст персонализации. Голод, время и последний приём
пищи всегда выясняются из текущего диалога. Пока пользователь явно не подтвердил
`last_meal_type=small_snack`, ассистент использует нейтральное выражение «приём
пищи», а не «перекус».

API-ключ и ID каталога хранятся отдельно в защищённом хранилище Android/iOS. В MVP
запросы идут с телефона напрямую в Yandex AI Studio, поэтому пользователь вводит
собственные реквизиты и оплачивает использование модели в своём Yandex Cloud.

## Что нужно сделать в Yandex AI Studio

1. Войти в [Yandex Cloud](https://console.yandex.cloud/) и создать облако и каталог,
   если их ещё нет. Для платных API должен быть активен платёжный аккаунт.
2. Открыть [Yandex AI Studio](https://aistudio.yandex.ru/).
3. Нажать **«Создать API-ключ»** в правом верхнем углу, выбрать срок действия и
   сохранить **секретное значение ключа**. После закрытия окна оно повторно не
   показывается.
4. Скопировать **ID каталога**: в верхней части AI Studio навести указатель на
   название текущего каталога и нажать значок копирования.
5. При первом запуске приложения вставить две строки: секрет API-ключа и ID
   каталога. Приложение проверит доступ к Alice AI LLM Flash до сохранения.

Ключ, созданный через интерфейс AI Studio, автоматически получает минимально
необходимые права. Если ключ создаётся вручную для существующего сервисного
аккаунта, ему нужна роль `ai.languageModels.user` на каталог и scope
`yc.ai.foundationModels.execute`.

Официальные материалы:

- [создание API-ключа](https://aistudio.yandex.ru/docs/ru/ai-studio/operations/get-api-key.html);
- [доступные модели и URI](https://aistudio.yandex.ru/docs/ru/ai-studio/concepts/generation/models.html);

## Архитектура проекта

- `lib/src/agents/` — агенты, схемы структурированных ответов и валидаторы;
- `lib/src/services/alice_client.dart` — Alice AI и embeddings API;
- `lib/src/data/` — secure storage и SQLite/RAG;
- `lib/src/screens/` — онбординг и чат;
- `lib/src/widgets/` — glass-компоненты интерфейса.

Ключевые данные `CaseState`:

- `request` — исходное желание пользователя без потери формулировки;
- `facts` — только факты текущего диалога;
- `hypotheses` — проверяемые предположения агентов;
- `missing_fields` и `asked_fields` — план и история уточнений;
- `relevant_history` — отдельный RAG-контекст, не смешиваемый с `facts`;
- `rejected_advice` — варианты, с которыми пользователь не согласился.

## Безопасность рекомендаций

Приложение не ставит диагнозы и не заменяет врача. Детерминированные правила не
позволяют модели:

- предлагать паузу вместо еды при выраженном физическом голоде;
- считать лёгкий приём пищи полноценным и игнорировать длительный перерыв;
- рекомендовать голодание как компенсацию переедания;
- игнорировать аллергию, диабет, головокружение или подготовку к анализам;
- разрешать энергетик как автоматический запасной вариант;
- трактовать уже съеденную еду и чувство стыда как новый запрос на перекус;
- называть неизвестный прошлый приём пищи «перекусом»;
- оставлять повторяющуюся потерю контроля без предложения поддержки специалиста.

Уточняющие вопросы задаются по одному, не повторяются и ограничены пятью на
сессию. Осознанное социальное удовольствие, например общий десерт на свидании,
не объявляется проблемой автоматически.

## Ограничения MVP

Это инструмент самонаблюдения, а не медицинский продукт и не замена врачу. В
текущей BYOK-архитектуре ключ защищён системным хранилищем устройства, но запросы
всё равно отправляются напрямую из приложения. Перед публикацией в стор стоит
добавить собственный backend, удалённую конфигурацию политик, шифрование локальной
базы и экспертную проверку сценариев, связанных с расстройствами пищевого
поведения.
